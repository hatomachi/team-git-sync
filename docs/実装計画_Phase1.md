# Team Git Sync: 実装計画 (Implementation Plan)

> **注記:** 本ドキュメントは、AIアシスタントとのセッションをまたいでも実装の前提や計画を参照できるように、`docs` フォルダに保存・更新されるインベントリです。

## 重要な前提条件
VaultのルートディレクトリがGitリポジトリなのではなく、**Vault直下の各フォルダ（例: `teamA/`, `teamB/`）がそれぞれ独立したGitリポジトリである**という構造を前提とします。

これにより、プラグイン側で行う `simple-git` の初期化プロセスは、「Vaultパス」ではなく「**現在アクティブなファイルが属するサブフォルダ（Gitルート）**」を動的に判定・指定する必要があります。

---

## Phase 1: プラグインの土台作りとGit連携テスト

「方針.md」の定義に従い、Obsidianプラグインの基本構成を作成し、Node.js環境から `simple-git` を用いてVault内の各リポジトリにアクセスできることを確認します。

### プラグインの基本構成 (完了済)
* `manifest.json`: メタデータ
* `package.json`: 依存関係（`simple-git`, `builtin-modules` など）とビルドスクリプト
* `tsconfig.json`: TypeScript設定
* `esbuild.config.mjs`: `main.js` のバンドル設定（Node.js組み込みモジュールの外部化対応済）

### メインロジック (次に実装するステップ)
* `main.ts`: プラグインのエントリーポイント
  * `simple-git` の初期化：アクティブなファイルが変更されたタイミングで、そのファイルの親ディレクトリを遡り `.git` フォルダが含まれるディレクトリを特定する（またはVault直下の第1階層フォルダをリポジトリとみなす等）ロジックを実装。
  * テストコマンド「Test Git Status」の改良：単にVaultルートを見るのではなく、上記で特定したGitリポジトリのパスに対して `git.status()` を実行し、結果をコンソール出力・Notice表示する。

### 確認手順
1. Vault直下にテスト用のフォルダ（例: `teamA`）を作成し、その中で `git init` を実行する。
2. そのフォルダ内で適当なファイルを作成する。
3. プラグインを起動し、作成したファイルを開いた状態で「Test Git Status」コマンドを実行する。
4. 正しく対象ディレクトリが認識され、ステータスが表示されるか確認する。
